# Glyph User-Facing Language TODOs

This file tracks post-MVP changes to the author-facing Glyph surface. The bias is to keep source files small and readable: add syntax only when it captures author intent that the compiler cannot infer reliably.

## Guiding Principle

Minimize user-facing complexity without losing semantic value. If a concept can be inferred from existing source structure, keep it out of the authoring language and expose it through IR, manifest, risk analysis, visualization, or search tooling instead.

## Proposed Surface Delta

Add two optional sections:

```glyph
skill fix_bug(scope = ".") -> BugFixReport
    description: "Use when the user reports a concrete bug or regression."
    goal: "Find the root cause and make the smallest safe change that fixes it."

    constraints:
        require preserve_existing_patterns
        avoid unrelated_edits

    output:
        require summary
        require root_cause
        require files_changed
        require verification
        require remaining_risks
        must avoid fabricated_test_results

    flow:
        ctx = inspect_repo(scope)
        failure = reproduce_failure(ctx)
        fix = implement_targeted_fix(failure)
        verify_fix(fix)
        return <bug_fix_report>
```

## Section Semantics

- `description:` remains routing metadata: when an agent should select the skill.
- `goal:` states execution intent: what successful execution means once the skill is selected.
- `context:` remains non-normative background.
- `constraints:` remain process guardrails: how work must or should be done.
- `output:` defines the result contract: what the final response or artifact must contain, may contain, or avoid.
- `flow:` remains the ordered procedure.

`goal:` should not compile as a workflow step or behavioral constraint. It should project as goal/intention in compiled output and as searchable intent metadata in any manifest representation.

## Output Contract Model

Keep `-> DomainType` as the nominal result kind. Use `output:` for the richer contract.

```glyph
skill research(topic) -> ResearchBrief
    output:
        require key_findings
        require tradeoffs
        require sources
        require open_questions
```

The same marker vocabulary should be valid in `output:`:

- `require`
- `avoid`
- `must`
- `must avoid`

These markers produce `OutputContract` clauses, not normal behavioral constraints. For example:

```glyph
constraints:
    must avoid fabricated_test_results
```

means the agent must not fabricate while doing the work.

```glyph
output:
    must avoid fabricated_test_results
```

means the final answer must not claim verification that did not happen.

## Return Relationship

Keep terminal `return` as the dataflow link between `flow:` and the output contract:

```glyph
return <bug_fix_report>
```

The return target names the agent-synthesized output. The return type says what kind of thing it is. The `output:` section says what the thing must contain or satisfy.

## Inferred, Not Authored

Do not add these as user-facing keywords:

- `phase:`
- `scene:`
- `action:`
- `resource:`
- `risk:`
- `intent_signature:`
- `scheduling:`
- `structural:`
- `logical:`

Those should be inferred from `description:`, `goal:`, parameters, `flow:`, `constraints:`, effects, and `output:`. Expose them through compiler-side artifacts such as `glyph inspect`, manifest JSON, visualizers, search indexes, and risk reports.

## Compiled Output

Restore compiled goal/output sections only when authored:

- `### Goal` or equivalent when `goal:` is present.
- `### Output` / `## Output` when `output:` is present and richer than a folded return sentence.

Open question: whether `goal:` belongs inside `## Instructions` as `### Goal` or near the top as a peer to parameters. The answer may depend on whether target agents use it better as execution guidance or routing metadata.

## Manifest And Tooling Implications

The paper-inspired SSL-like representation should be derived, not authored. With `goal:` and `output:`, the compiler can produce richer metadata for:

- skill retrieval and selection,
- workflow graph visualization,
- risk review,
- registry indexing,
- output validation,
- comparison between similar skills.

Design rule: every derived manifest claim should carry provenance back to source spans or IR node IDs.

## Caveats and Considerations

The proposals above are the current direction. Several alternative or complementary approaches were discussed that may shift these decisions:

### `description:` as compiler-generated by default

The compiler already generates `description:` in the repair pass when it's absent. Could make this the default behavior: the author only writes `description:` when the generated version is wrong or imprecise. This reduces the minimum viable skill surface — one less section a novice author needs to think about.

**Paper data caveat:** The SSL paper's 0.573 MRR baseline uses the `description` field from 6,184 real, published SKILL.md files — these are **human-authored** descriptions, not LLM-generated ones. The paper does not benchmark auto-generated descriptions at all, so there is no evidence that a compiler-generated description would match this MRR. A generated description could plausibly be worse, which weakens the case for generation-by-default. The safer path may be to keep `description:` author-written as currently proposed, with compiler generation remaining the repair-pass fallback (as it is today) rather than the primary path.

**Retrieval data for context** (all against hand-authored descriptions):

| Input | MRR | Correct skill at ~rank |
|---|---|---|
| Desc_only (human-authored one-liner) | 0.573 | 1.75 |
| Full SKILL.md (entire prose doc) | 0.602 | 1.66 |
| Desc + SSL-Rich (structured metadata) | 0.707 | 1.41 |

Key takeaway: the big retrieval gains come from structured metadata (+0.105 MRR from SSL over Desc_only), not from richer prose (+0.029 from Full SKILL.md over Desc_only). So the author's `description:` is worth keeping for the baseline 0.57, but the real routing improvement will come from the compiler-derived manifest — params, return type, effects, scene structure — not from trying to generate a better one-liner.

### `output:` as compiler-generated by default

The descriptive return (`return <"root cause analysis including affected files and severity">`) already names required output fields in prose. The compiler could extract those into structured `OutputContract` clauses, making `output:` an author-override rather than an author-write. This keeps the novice kernel lean — one `return` line covers both dataflow and output contract — while giving power authors a way to tighten the contract explicitly. The tradeoff is that prose extraction may miss fields or produce an incomplete contract, and the author has no structured way to say "this field is optional" until structural types exist.

### `goal:` is not inferrable (author essential)

Unlike `description:` and `output:`, `goal:` captures genuine author intent about success criteria — what "done" means for this skill. The compiler can guess from the skill name and body, but this is the kind of signal that routes agent behaviour downstream, and guessing it wrong is worse than leaving it absent. Of the three proposed sections, `goal:` is the strongest candidate for always author-written.

### `effects:` killed from the author surface

The compiler infers effects from the call graph. The author-facing `effects:` section (currently gated behind `--enable-effects`) could be removed entirely — effects live in IR and manifest/JSON output only. This is consistent with the "Inferred, Not Authored" principle and removes one more thing from the author's plate. The tradeoff is losing the author's ability to assert `effects: none` as a contract; but that assertion could move to the `glyph check` tooling layer instead.

### Generation defaults summary

If all of the above were adopted, the section responsibilities would be:

| Section | Default behavior | Author writes when |
|---|---|---|
| `description:` | Generated from skill name + body | Generated version is wrong or imprecise |
| `goal:` | Absent (no generation) | Author wants to pin success criteria |
| `output:` | Generated from `return <"...">` prose | Generated contract is incomplete or too loose |
| `constraints:` | Absent (no generation) | Author has behavioral rules to declare |
| `context:` | Absent (no generation) | Author has background to share |
| `flow:` | Required (no generation) | Always — the author's procedure |
| `effects:` | Inferred from call graph, surfaced in IR/manifest only | Never — removed from author surface |

Principle: the author writes what the compiler can't know (procedure, constraints, intent). The compiler generates what it can derive (description, output contract, effects, scene structure, resource scopes).

## Deferred Parser Support

Items where the language guide / design already commits to a syntax but the MVP parser does not yet accept it. These are implementation gaps, not open design questions.

### Typed parameters

`GLYPH_LANGUAGE_GUIDE.md` §"Type annotations" and `design/types.md` describe parameter type annotations of the form `name: DomainType` (e.g. `skill implement_feature(scope: PathSpec, risk: RiskLevel = "medium")`). The MVP parser's `parse_param_list` (`crates/glyph-core/src/parse.rs`) only accepts `name [= "default"]` — it stops at the `:`, leaving the parser to fail on the next `expect(Rparen)`.

When the failing source also contains a descriptive return (`return <"…">`), the post-parse `<`-scan formerly masked the real cause by emitting `G::parse::output-target-outside-return` against the unconsumed `<` — pointing the author at the wrong line entirely. The scan is now gated on the parser's failure offset: only `<` tokens at-or-before the failure are reported (which preserves the structured diagnostic for stray-`<` cases like `< bar` at statement start, where the `<` itself *is* the failure cause); tokens past the failure are unreached and suppressed. With typed-param parsing still missing, the parameter-list failure surfaces as a generic `Parse(Eof)` instead of a misdirected output-target diagnostic.

Landing typed parameters requires:

- Extending `parse_param_list` to optionally consume `: <Ident>` between the name and any default.
- Adding `type_annot: Option<Spanned<String>>` to the `Param` AST node (and forwarding through `IrParam` if the IR consumers want it; today `analyze.rs::emit_nominal_mismatch` is already a placeholder waiting for typed annotations to land).
- Regression tests covering `name: Type`, `name: Type = "default"`, and the negative `name:` (missing type ident).

Until then, authors must omit type annotations on parameters even though the language guide treats them as part of MVP.

## Open Questions

- Should the source keyword be singular `output:` or plural `outputs:`? Prefer `output:` unless multi-output values become real.
- Should `output:` allow only marker clauses, or also typed field declarations later?
- Should `goal:` allow multiple entries, or exactly one string/bare const like `description:`?
- Should blocks and export blocks support `goal:` and `output:`, or only skills at first?
- Should output clauses support `optional`, or should optionality stay in prose until structural types exist?

## Known Author-Visible Emitter Gaps

### Branch-as-last-step drops `return <X>` suffix (P1 from issue #118 codex review)

**Symptom for the author:** A skill or block whose `flow:` ends with a structural step (e.g. `if/elif/else`) followed by `return <X>` compiles with no "as your result" prose anywhere in the rendered Markdown — the author's output target is silently dropped.

**Reproduction:**

```glyph
skill main()
    description: "..."
    flow:
        if condition
            "do A"
        else
            "do B"
        return <current_branch>
```

The compiled `.md` shows the `if … then:` block but never folds `current_branch` into a final visible step. The four locked OC templates (`append_identifier_suffix`, `append_description_suffix`, `standalone_return_identifier`, `standalone_return_description`) never fire.

**Root cause:** `crates/glyph-core/src/emit/scaffold.rs` (the `IrNode::Branch` arm of the main scaffold walker) ignores `is_last` and `skill_oc_form`. The `IrNode::InlineInstruction` arm and the `IrNode::Call` Tier-1 arm both honor these and route through the locked templates; the branch arm does not.

**Recommended fix shape:** When the last visible step is `IrNode::Branch` and `skill_oc_form.is_some()`, emit the branch as today, then append one additional standalone return step using `templates::standalone_return_identifier(name)` or `templates::standalone_return_description(desc)`. This mirrors how the empty-resolved-body Tier-1 path already handles the case where there is no body to fold the suffix into. Folding the suffix into the last sub-step of the last branch arm is the alternative but it's awkward — different arms produce different terminal sub-steps and the suffix would need to fold into each independently; the design doc is silent on this, and the standalone-after-branch shape is the least surprising and most consistent with existing Tier-1 callees.

**Audit candidates** (same gap likely recurs):

- Tier-2 same-file procedure path (`### Procedure:` block, `projection_tier == 2`) when a branch is the final step inside the procedure body.
- Tier-3 external-file path (`emit_procedure` in `emit/mod.rs`) — same question.
- Block-level emission when an `IrNode::Block` is materialized as a procedure — does its emitter also gate on `is_last` for branches?

**Test gap:** No fixture currently has `if/else/return <X>` as the closing pattern, which is why this regression has not been caught. A regression test exercising both `return <Identifier>` and `return <"description">` shapes with a terminal branch should accompany the fix.
