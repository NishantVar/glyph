# Glyph Block Structure And Indentation

This document records the MVP decisions for how Glyph source delimits blocks, introduces sub-sections, handles indentation, and relates to Markdown.

## Decision: Significant Indentation

Glyph uses Python-style significant indentation. There are no braces, no `end` keywords, and no other explicit block delimiters.

A block is the set of contiguous lines indented deeper than the introducing line. A block closes when the next non-blank, non-comment line appears at equal or lesser indentation.

Justification: principle 4 (Python-like readability), principle 1 (hierarchy obvious at a glance), principle 7 (small core language). Braces add punctuation noise. Keyword delimiters add verbosity. Neither serves Glyph's goals.

## Indentation Unit

The indentation unit is 4 spaces.

- Tabs are rejected as a compile error.
- Mixed indentation (tabs and spaces on the same line) is a compile error.
- The LLM repair pass may auto-fix tab-indented source to 4 spaces.
- Indentation must increase by exactly 4 spaces per nesting level. Other increments are a compile error.

## Sub-Section Headers

Sub-sections within a declaration body are introduced by a colon-terminated keyword. The colon is what distinguishes a labeled sub-block from a bare instruction or declaration header.

MVP sub-section keywords: `flow:`, `effects:`, `inputs:`, `outputs:`, `constraints:`, `when_to_use:`.

Two forms are allowed:

**Long form** — keyword on its own line, indented body below:

```glyph
effects:
    - reads_files
    - writes_files
    - runs_commands
```

**Short form** — keyword and content on the same line:

```glyph
effects: reads_files, runs_commands
```

The compiler normalizes both forms to the same IR representation.

## Top-Level Declarations

Top-level declarations (`skill`, `block`, `export block`, `text`, `export text`, `import`) introduce their body by indentation alone. They do not use trailing colons. The colon-terminated form is reserved for sub-section headers within declaration bodies.

This creates a visual distinction: declarations are structural headers; colons introduce labeled sub-blocks within them.

Whether top-level declaration headers use other trailing syntax (such as parameter lists or `=`) is owned by the declaration-header design, not by block structure.

## Blank Lines

Blank lines inside blocks are freely allowed and do not close or break blocks. They are visual separators only.

A block continues until the next non-blank, non-comment line at equal or lesser indentation. Authors may use blank lines to group related instructions visually without affecting structure.

## Line Continuation

Line continuation is implicit inside paired delimiters only. There is no explicit continuation character.

- Parenthesized expressions (call arguments) may span multiple lines until the closing `)`.
- Braced import lists (`{ ... }`) may span multiple lines until the closing `}`.
- Triple-quoted block strings (`"""`) span lines until the closing `"""`.

No backslash continuation. This keeps the language simple (principle 7).

Examples:

```glyph
import "./coding_agent_safety.glyph.md" {
    unrelated_edits,
    preserve_existing_patterns as existing_patterns,
    validate_before_success,
}

plan = make_plan(
    ctx,
    risk,
    strict = true,
)
```

Inside paired delimiters, indentation is not structurally significant. The logical line is not complete until delimiters balance.

## Markdown Coexistence

The entire `.glyph.md` file is Glyph source. There is no Markdown passthrough.

The `.glyph.md` extension signals "this is a Glyph source file" and enables the `.glyph.md` to `.md` compilation convention (boundary 5). Markdown structure lives in the compiled output, not in the source.

Inline instruction strings (inside `"..."` or `"""..."""`) may contain Markdown-like text, but the parser treats them as opaque string content. The Markdown headings, lists, and formatting in the compiled `.md` file are generated by the compiler from IR roles and the fixed section order defined in [compiled-output.md](compiled-output.md). Authors provide content and Glyph structure; the compiler provides the Markdown skeleton.

## Nesting Levels

Glyph source uses three primary indentation levels in practice:

- **Level 0** (column 0): top-level declarations (`skill`, `block`, `text`, `import`).
- **Level 1** (column 4): declaration body — constraints, sub-section headers, bare instructions.
- **Level 2** (column 8): sub-block body — flow steps, effect list items, nested content.

Deeper nesting is structurally supported for constructs such as `if` inside `flow:`, or nested private blocks inside `skill`. The exact nesting rules for control flow are deferred to later design.

## Complete Example

```glyph
import "./repo_tools.glyph.md" {
    unrelated_edits,
    preserve_existing_patterns,
}

text validate_before_success = """
    Run the full validation suite and confirm all checks pass
    before reporting success.
"""

skill implement_feature(scope, risk = "medium")
    require preserve_existing_patterns
    avoid unrelated_edits
    prefer simple_solution

    effects:
        - reads_files
        - reads_env
        - writes_files
        - runs_commands

    flow:
        ctx = inspect_repo(scope)
        plan = make_plan(ctx, risk)
        apply_changes(plan)
        validate(plan)
        return summarize(plan)
```

## Interaction With Comment Syntax

Comment-only lines are invisible to the indentation parser. A comment on its own line does not affect block structure. Trailing comments on code lines do not affect indentation measurement — indentation is determined by leading whitespace before code content.

## Interaction With Declaration Headers

Block structure requires that declaration headers introduce their body through indentation. It does not prescribe the shape of the header line itself (parameter syntax, return type annotations, etc.). The declaration-header design owns that surface.

## Interaction With LLM Repair

The repair pass may fix malformed indentation by normalizing to 4-space levels. Tab-to-space conversion and indentation correction are accepted repair actions per [llm-repair-pass.md](llm-repair-pass.md).

## Deferred Sub-Questions

- Whether `if` uses a colon (`if condition:`) or introduces its body by indentation alone. Deferred to control-flow syntax design.
- Whether `for_each` (post-MVP) uses a colon. Deferred.
- Whether nested private blocks inside `skill` create nested indentation scopes or follow different rules. Deferred to later design but structurally supported by significant indentation.
