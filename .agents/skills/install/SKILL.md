---
name: install
description: 'Install Glyph by downloading prebuilt artifacts from the latest GitHub release (Core CLI/LSP, Editor Extensions, and Agent Skills/Commands), falling back to source builds when downloads are unavailable.'
---

## Parameters

- **repo_root**. Default: ".". Absolute or relative path to a local Glyph clone, used when source-build fallbacks need to find `Cargo.toml`, the editor extension sources, or the agent skills tree.

## Context

- **installation-phases**

  The installation has three phases: Core CLI/LSP binary, Editor extension (.vsix), and Agent integration (skills + commands). Each phase tries the prebuilt release artifact first and falls back to a source build only if the download cannot be used.

- **release-first-strategy**

  Default to downloading prebuilt artifacts from the latest GitHub release (https://github.com/NishantVar/glyph/releases). Build from source only when a release asset is missing for the user's platform, the download fails, or the user explicitly opts to build.

- **optional-node-deps**

  Node and npm are only needed for the source-build fallback of the editor extension. They are not required when the latest release ships a prebuilt .vsix. The core CLI install must proceed regardless of whether node/npm are present.

## Constraints

- **Must:** Never install missing prerequisites silently. Always ask the user for explicit permission to auto-install missing tools (e.g., via rustup, brew, or apt) before running any installer.
- **Must:** Do not block the core CLI install or the agent-skills install if node/npm are missing and the user declines to install them. Skip only the editor extension phase in that case.
- **Require:** Prefer prebuilt release artifacts over source builds whenever a matching asset exists for the user's platform. Source builds are a documented fallback, not the default.

## Steps

1. Default path — follow the install-cli-binary procedure below to install the core Glyph CLI/LSP binary, preferring a prebuilt release archive for the host platform and falling back to a `cargo install` from `{repo_root}` only when the download path cannot be used.
2. Default path — install the VS Code-compatible editor extension by downloading the prebuilt `.vsix` from the latest GitHub release. List the assets on the latest GitHub release (prefer `gh release view --repo NishantVar/glyph --json assets -q '.assets[].name'`, otherwise `curl` the API); if a `.vsix` asset is published, download it to a temp directory using the same tooling as the CLI install. Probe for VS Code (`code`), Cursor (`cursor`), Windsurf (`windsurf`), Antigravity (`antigravity`), and VSCodium (`codium`), including OS-specific paths like `/Applications/Visual Studio Code.app/Contents/Resources/app/bin/code` on macOS. Ask the user which detected editors to install into, then for each chosen editor run `<cli> --uninstall-extension glyph.glyph-language` (ignore errors) followed by `<cli> --install-extension <vsix_path>`. As a fallback when no `.vsix` is published, the download fails, or the user opts to build, verify the source is at `{repo_root}/editors/vscode/package.json` (otherwise instruct the user to clone https://github.com/NishantVar/glyph and rerun with the correct `repo_root`, then skip this phase). Check `command -v node` and `command -v npm` and ask permission to install Node.js (brew on macOS, apt on Linux) if missing; if the user declines or installation fails, explicitly state the editor extension build is being skipped and exit this phase so the rest of the workflow continues. With node and npm available, probe and choose editors as in the default path, then `cd "{repo_root}/editors/vscode"`, run `npm install` and `npm run package` to produce the `.vsix`, and install it into each chosen editor (uninstalling the previous version first).
3. Run `command -v glyph` to locate the installed binary (which doubles as the LSP server). If it resolves, print the absolute path so the user can confirm; if it does not resolve, the binary is installed somewhere not on PATH (for example `~/.local/bin/glyph` when `~/.local/bin` is not exported, or `~/.cargo/bin/glyph` from a fallback source build). Instruct the user to either add that directory to their shell PATH or set `glyph.serverPath` in each editor's `settings.json` to the absolute path of the binary.
4. Check the user's home directory for configuration directories of popular coding agents — Gemini CLI at `~/.gemini/`, Claude Code at `~/.claude/`, Codex at `~/.codex/`, Goose at `~/.config/goose/` or a similar standard path, and OpenCode / OpenHands — and refer to the resulting set of paths as agent_dirs.

   Output: the list of detected agent configuration directories.
5. Print the detected configuration directories from the agent_dirs identified in step 4, ask the user which of those agents to install the Glyph skills and commands into, and accept either comma-separated names, "all", or an empty response (defaulting to all). Refer to this result as chosen_agents.

   Output: the agent directories the user chose to install into.
6. Ask the user how they want to install the optional Glyph commands: with descriptions (automatic routing), without descriptions (manual routing only, saves context), or not at all. Map their choice to one of the string values `"with_desc"`, `"no_desc"`, or `"none"`, and refer to it as commands_choice.

   Output: the chosen commands preference, one of with_desc, no_desc, or none.
7. Resolve the absolute path to a Glyph source tree to install skills from; the install mode is inferred from whether the user already has a local Glyph clone, so no separate prompt is asked. If `{repo_root}` already points at a Glyph clone — verified by checking for `{repo_root}/.agents/skills/glyph/SKILL.md` and `{repo_root}/scripts/install_agent_skills.sh` — use that absolute path, since the caller will symlink agent configs into it and the user is expected to keep the clone on disk and `git pull` it to upgrade. Otherwise, create a temporary working directory via `tmp=$(mktemp -d)`, run `git clone --depth 1 https://github.com/NishantVar/glyph.git "$tmp/glyph"`, and use `$tmp/glyph` as the resolved path; the caller will copy from it and the temp directory may be removed afterward. Never create a persistent clone in a hidden path like `~/.local/share/glyph` — that would leave state the user did not ask for. Refer to this result as skills_source.

   Output: the absolute path to the resolved Glyph source tree to install skills from.
8. Iterate over each agent directory in the chosen_agents from step 5 and infer the install mode from the skills_source resolved in step 7: if its path begins with `/tmp/` or `/var/folders/`, use copy mode; otherwise, use symlink mode because the user brought their own Glyph clone and we will link into it. In symlink mode, run `"<skills_source>/scripts/install_agent_skills.sh" "<agent_dir>" "<commands_choice>"` — the script creates the `skills` and `commands` subdirectories if missing and creates symlinks pointing back into the skills_source — and capture and display its output. In copy mode, for each agent directory ensure `<agent_dir>/skills/` and `<agent_dir>/commands/` exist via `mkdir -p`, replace any existing destination first with `rm -rf "<agent_dir>/skills/glyph"` and (when applicable) `rm -rf "<agent_dir>/commands/glyph"`, then copy the core skill with `cp -R "<skills_source>/.agents/skills/glyph" "<agent_dir>/skills/glyph"`; branch on the commands_choice from step 6 so that `"with_desc"` copies `<skills_source>/.agents/commands/glyph` into `<agent_dir>/commands/glyph`, `"no_desc"` copies `<skills_source>/.agents/commands_no_desc/glyph` into the same destination, and `"none"` skips the commands copy entirely; finally print which paths were written for that agent. After processing all agents, if the inferred mode was copy, tell the user it is safe to remove the temp directory at the skills_source from step 7, then provide a concise summary of the installation results across all phases, noting whether each phase used a downloaded release artifact or fell back to a source build.
9. After the summary, follow the show-post-install-guidance procedure below to list the entry points the user can now start using, with phrasing chosen based on the commands_choice captured in step 6.

### Procedure: install-cli-binary

1. By default, download the prebuilt CLI binary from the latest GitHub release. Detect the host platform with `uname -s` and `uname -m` and map it to one of the published target triples — Darwin/arm64 to `aarch64-apple-darwin`, Darwin/x86_64 to `x86_64-apple-darwin`, Linux/x86_64 to `x86_64-unknown-linux-gnu`, Windows/x86_64 (MSYS or Git Bash) to `x86_64-pc-windows-gnu` — skipping straight to the fallback if no triple matches. Resolve the latest release tag using `gh release view --repo NishantVar/glyph --json tagName -q .tagName` when `gh` is available, otherwise parse the value from `curl -fsSL https://api.github.com/repos/NishantVar/glyph/releases/latest | grep '"tag_name"'`. Download the matching archive `glyph-<tag>-<target>.tar.gz` (or `.zip` on Windows) into a temp directory, preferring `gh release download <tag> --repo NishantVar/glyph --pattern '<archive>' --dir <tmp>` or falling back to `curl -fsSL -o <tmp>/<archive> https://github.com/NishantVar/glyph/releases/download/<tag>/<archive>`. Extract the archive, move the `glyph` binary (`glyph.exe` on Windows) into `~/.local/bin/` (creating the directory with `mkdir -p ~/.local/bin` if needed), make it executable with `chmod +x ~/.local/bin/glyph`, and verify with `~/.local/bin/glyph --version` (or `glyph --version` if it is already on PATH); if `~/.local/bin` is not on the user's PATH, instruct them to add `export PATH="$HOME/.local/bin:$PATH"` to their shell rc file. As a fallback when the download path failed (no matching asset, network failure, unsupported platform) or the user explicitly opts to build, verify the source repo is present at `{repo_root}` by checking for `{repo_root}/Cargo.toml` and `{repo_root}/crates/glyph-cli` (otherwise ask the user to clone https://github.com/NishantVar/glyph and rerun or supply the correct `repo_root`), then run `command -v cargo` to verify Cargo and ask for explicit permission to install Rust/Cargo via rustup if missing — halting with a clear message that the CLI cannot be installed without either a working release download or Cargo if the user declines — and finally run `cargo install --path crates/glyph-cli` from `{repo_root}`.

### Procedure: show-post-install-guidance

1. After the summary, list the Glyph entry points the user can now start using, in priority order; the exact phrasing depends on the commands_choice from step 6. When commands_choice is `"with_desc"` or `"no_desc"` and slash commands were installed, present these four in this order — `/glyph:decompile` to convert any existing compiled `.md` skill into a `.glyph` source (most useful for reading skills, since `.glyph` is dramatically more readable than the prose form so browsing skills feels more like reading code), `/glyph:teach` to author or edit a `.glyph` skill from scratch or from a task description (the primary entry point for writing new skills), `/glyph:compile` to compile a `.glyph` source into the agent-consumable `.md` form, and `/glyph:icompile` to make a targeted edit to a `.glyph` and its paired compiled `.md` in lockstep without re-running the full compile pipeline — and close with a note that the user can also just invoke the `glyph` skill (or `/glyph`) with whatever they want to do and it will route to the right sub-skill. When commands_choice is `"none"` and no slash commands were installed, present the same four in the same order but as skill names, telling the user to ask their agent to invoke each by name: `glyph:decompile` (same purpose as above, invoked by asking the agent to use the `glyph:decompile` skill on a target `.md`), `glyph:teach` (invoked by asking the agent to use the `glyph:teach` skill when writing or editing a `.glyph` source), `glyph:compile` (invoked by asking the agent to use the `glyph:compile` skill on a `.glyph` source), and `glyph:icompile` (invoked by asking the agent to use the `glyph:icompile` skill for paired source-and-compiled edits) — and close with the same note that the user can also invoke the top-level `glyph` skill with whatever they want to do and it will route to the right sub-skill.
