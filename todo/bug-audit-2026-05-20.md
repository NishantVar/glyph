## B10 - Imported Tier-1 callee output contracts are lost in emitted IR JSON

Severity: medium-high. Downstream tools reading emitted IR cannot see the output contract of imported calls even though the compiler had that metadata.

Repro:

```sh
mkdir -p tmp/bug-audit-imported-oc
printf '%s\n' \
  'export block helper() -> BranchName' \
  '    flow:' \
  '        return <current_branch>' \
  > tmp/bug-audit-imported-oc/lib.glyph
printf '%s\n' \
  'import "./lib.glyph" { helper }' \
  '' \
  'skill main() -> BranchName' \
  '    description: "Main."' \
  '    flow:' \
  '        helper()' \
  '        return <current_branch>' \
  > tmp/bug-audit-imported-oc/main.glyph

target/debug/glyph compile tmp/bug-audit-imported-oc/main.glyph --emit-ir --out-dir tmp/bug-audit-imported-oc/out --format json
rg -n 'callee_output_contract|current_branch|helper' tmp/bug-audit-imported-oc/out/main.ir.json
rm -rf tmp/bug-audit-imported-oc
```

Observed:

- The `helper` call emits `"callee_output_contract": null` in `main.ir.json`, even though the imported block returns `<current_branch>`.

Likely cause:

- `crates/glyph-core/src/emit_ir.rs` derives `callee_output_contract` by looking up a same-file `IrBlock`.
- Imported Tier-1 callees live only on `IrCall.callee_output_contract`, so the JSON path drops them.

Best fix:

- In `emit_ir.rs`, prefer the contract already attached to `IrCall`.
- Fall back to arena/block lookup only when the call-level contract is absent.
- Add an imported-callee IR JSON regression test.

This bug is also listed in the existing bug inventory, but the repro above verifies it in the current worktree.

Severity: medium. Users and agents cannot rely on the diagnostic contract docs.

## B14 - LSP and semantic-token coordinate conversion is likely wrong for UTF-16

Severity: medium. LSP uses UTF-16 character offsets by default, but Glyph spans and semantic tokens are byte-column based.

Evidence:

- `crates/glyph-lsp/src/convert.rs` maps Glyph byte columns directly to LSP `Position.character`.
- `crates/glyph-core/src/semantic_tokens.rs` notes that semantic token positions are byte columns and non-ASCII string literal contents may be off.
- `LineIndex::byte_offset` in `crates/glyph-core/src/span.rs` does not appear to clamp columns to line length despite its doc comment, so out-of-line cursor columns may map into following lines.

Best fix:

- Add UTF-16-aware position conversion against the source text for diagnostics, definitions, hover, and semantic tokens.
- Update `LineIndex::byte_offset` to clamp to the requested line's end.
- Add tests with non-ASCII characters before a referenced token and with cursor positions beyond line end.
