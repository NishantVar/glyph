## B02 - Documented `!=` conditions are not implemented

Severity: high. The public language guide documents inequality, but the compiler and tree-sitter grammar reject it.

Repro:

```sh
mkdir -p tmp
printf '%s\n' \
  'const low = "low"' \
  '' \
  'skill main(risk = "high")' \
  '    description: "Demo."' \
  '    flow:' \
  '        if risk != "low":' \
  '            "Act."' \
  > tmp/bug-audit-neq.glyph

target/debug/glyph check tmp/bug-audit-neq.glyph --format json
target/debug/glyph compile tmp/bug-audit-neq.glyph --output tmp/bug-audit-neq.md --format json
rm -f tmp/bug-audit-neq.glyph tmp/bug-audit-neq.md
```

Observed (post-B01):

- `glyph check` emits `G::parse::unexpected-char` for the `!` of `!=` and exits 1.
- `glyph compile` does the same.
- The diagnostic is misleading because `!=` is a documented operator; the user sees a "not part of any Glyph token" message instead of an inequality-not-supported message. The underlying bug — `!=` is not implemented despite being in the language guide — is unchanged from before B01.

Evidence:

- `GLYPH_LANGUAGE_GUIDE.md` documents both `risk == "high"` and `risk != "low"`.
- `design/data-flow.md` also lists inequality.
- The tokenizer has `DoubleEquals` but no `NotEquals`.
- Condition parsing/classification handles `==`, but not `!=`.
- `tree-sitter-glyph/grammar.js` also only models `==`.

Best fix:

- Add `TokenKind::NotEquals`.
- Parse and classify `!=` wherever `==` is currently supported.
- Update diagnostics spans for both operands.
- Update `tree-sitter-glyph/grammar.js` and corpus tests.
- If inequality is intentionally out of scope, remove it from public docs instead, but that would be a language contract change.

## B03 - Export block flow is not fully semantic-checked

Severity: high. Invalid calls, missing args, nominal type mismatches, invalid body markers, and invalid return placement can pass `glyph check` inside `export block` declarations.

Representative repros, all of which currently produce no diagnostics. Write any one snippet to `tmp/export-block-repro.glyph`, then run `target/debug/glyph check tmp/export-block-repro.glyph --format json`.

Undefined return call:

```glyph
export block caller() -> Report
    flow:
        return missing()
```

Missing required arg:

```glyph
export block foo(x: Input) -> Report
    flow:
        return "ok"

export block caller() -> Report
    flow:
        return foo()
```

Nominal mismatch:

```glyph
export block foo() -> Plan
    flow:
        return "plan"

export block caller() -> Report
    flow:
        return foo()
```

Undefined body-level constraint:

```glyph
export block caller() -> Report
    require missing_text
    flow:
        return "ok"
```

Multiple returns:

```glyph
export block caller() -> Report
    flow:
        return "a"
        return "b"
```

Return not terminal:

```glyph
export block caller() -> Report
    flow:
        return "a"
        "after"
```

Return inside a branch:

```glyph
export block caller() -> Report
    flow:
        if "x":
            return "a"
        return "b"
```

Likely cause:

- `ExportBlockDecl` in `crates/glyph-core/src/ast.rs` stores flattened `flow_strings`, `body_refs`, and return metadata, but not a full `Vec<FlowStmt>`.
- `parse_with_diagnostics_opts` runs return-rule validation for `Skill` and `Block`, but not `ExportBlock`.
- `analyze_export_block` in `crates/glyph-core/src/analyze.rs` covers return type/missing-return/closure violations, but does not run the full call, name, arg, nominal return, branch-return, or body-marker validation used elsewhere.

Best fix:

- Prefer promoting export-block flow to the same structured representation used by normal blocks and skills.
- Run the same parser and analyzer validations for export blocks as for blocks/skills, with only the intentional export-block differences.
- Add regression tests for undefined calls, missing required args, nominal return mismatch, body marker resolution, multiple returns, return not terminal, and return inside branches.
- If a full AST migration is too large, add targeted export-block validation over `terminal_return`, `body_refs`, and call references as an interim patch.

## B04 - Pretty diagnostics point to the wrong columns

Severity: medium-high. JSON diagnostics carry the right columns, but human-facing pretty output points near column 2 for spans later in the line.

Repro:

```sh
target/debug/glyph check crates/glyph-cli/tests/corpus/invalid/case-violation/param_annotation_snake_case.glyph --format json
target/debug/glyph check crates/glyph-cli/tests/corpus/invalid/case-violation/param_annotation_snake_case.glyph --format pretty
```

Observed:

- JSON includes a type-case span at line 1, columns 14-22, and a return diagnostic at line 9, columns 16-18.
- Pretty output renders those diagnostics under column 2.

Likely cause:

- `locate_byte` in `crates/glyph-cli/src/main.rs` returns as soon as it reaches the requested line, before it advances to the requested column.

Best fix:

- Replace `locate_byte` with a correct implementation, ideally using `LineIndex::byte_offset` or an equivalent line-slice calculation.
- Add CLI tests that assert pretty caret placement for a diagnostic whose start column is not 1 or 2.

## B05 - Imported-file diagnostics are rendered against the wrong file and duplicated in directory checks

Severity: medium-high. This makes cross-file diagnostics misleading and noisy.

Single-entry repro:

```sh
mkdir -p tmp/bug-audit-diag-dep
printf '%s\n' \
  'skill dep()' \
  '    flow:' \
  '        "missing description"' \
  > tmp/bug-audit-diag-dep/dep.glyph
printf '%s\n' \
  'import "./dep.glyph" { dep }' \
  '' \
  'skill main()' \
  '    description: "Main."' \
  '    flow:' \
  '        "ok"' \
  > tmp/bug-audit-diag-dep/main.glyph

target/debug/glyph check tmp/bug-audit-diag-dep/main.glyph --format pretty
rm -rf tmp/bug-audit-diag-dep
```

Observed:

- The dependency file's missing-description diagnostic is rendered with the entry file path/source context.

Directory-mode repro:

```sh
mkdir -p tmp/bug-audit-diag-dep
printf '%s\n' \
  'export const dep_text = "x"' \
  'skill dep()' \
  '    flow:' \
  '        "missing description"' \
  > tmp/bug-audit-diag-dep/dep.glyph
printf '%s\n' \
  'import "./dep.glyph" { dep_text }' \
  '' \
  'skill main()' \
  '    description: "Main."' \
  '    require dep_text' \
  '    flow:' \
  '        "ok"' \
  > tmp/bug-audit-diag-dep/main.glyph

target/debug/glyph check tmp/bug-audit-diag-dep --format json
rm -rf tmp/bug-audit-diag-dep
```

Observed:

- The dependency diagnostic appears twice.

Likely cause:

- `run_check` in `crates/glyph-cli/src/main.rs` calls `check_file_with_effects`, receives a merged diagnostic bag, and renders all diagnostics against the entry file source/label.
- Directory mode repeats this for each file, so imported dependency diagnostics can be emitted once per importing root.
- JSON spans for dependencies only carry basenames such as `dep.glyph`, which is not enough path context in many directories.

Best fix:

- Have CLI check use `check_file_partition` or an equivalent API that preserves diagnostics by source path.
- Render each diagnostic with that file's own source and label.
- In directory mode, aggregate diagnostics by canonical path and diagnostic identity before rendering.
- Preserve enough relative or absolute path information in JSON spans for dependency diagnostics.

## B06 - `@glyph/std` imports do not resolve in the compile/directory pipeline

Severity: high for stdlib adoption. Ignored stdlib tests fail because imports like `import "@glyph/std" { subagent }` are still treated as missing imports or missing effects.

Evidence:

```sh
cargo test --workspace --no-fail-fast -- --ignored
```

Observed failures:

- `glyph-core --test callbodyshape_span::site_stdlib_toplevel_with_modifier_hard_fails` expected `G::expand::llm-required-for-call`, but got `G::analyze::stdlib-missing-import` and `G::analyze::missing-effects`.
- `glyph-core --test callbodyshape_span::trivial_stdlib_toplevel_renders_follow_procedure` expected a successful compile but got failed diagnostics.

Likely cause:

- `build_resolved_imports` in `crates/glyph-core/src/lib.rs` has a TODO and skips `@glyph/*`.
- A later compile-directory fast path also has a TODO and falls through to non-import-aware compile behavior, producing `stdlib-missing-import`.

Best fix:

- Carry stdlib effects and return metadata through the same path as file imports.
- Ensure stdlib-only consumers do not take the no-import compile fast path.
- Keep `load` rejected or gated according to the intended stdlib policy.

## B07 - Tree-sitter grammar rejects compiler-valid `with` modifiers

Severity: medium. The compiler accepts `with` modifiers on flow calls, but editor grammar parsing fails.

Observed:

Best fix:

- Update highlighting queries if modifiers should be highlighted specially.

## B10 - Imported Tier-1 callee output contracts are lost in emitted IR JSON

Severity: medium-high. Downstream tools reading emitted IR cannot see the output contract of imported calls even though the compiler had that metadata.

Repro:

```sh
mkdir -p tmp/bug-audit-imported-oc
printf '%s\n' \
  'export block helper() -> BranchName' \
  '    flow:' \
  '        return <current_branch>' \
  > tmp/bug-audit-imported-oc/lib.glyph
printf '%s\n' \
  'import "./lib.glyph" { helper }' \
  '' \
  'skill main() -> BranchName' \
  '    description: "Main."' \
  '    flow:' \
  '        helper()' \
  '        return <current_branch>' \
  > tmp/bug-audit-imported-oc/main.glyph

target/debug/glyph compile tmp/bug-audit-imported-oc/main.glyph --emit-ir --out-dir tmp/bug-audit-imported-oc/out --format json
rg -n 'callee_output_contract|current_branch|helper' tmp/bug-audit-imported-oc/out/main.ir.json
rm -rf tmp/bug-audit-imported-oc
```

Observed:

- The `helper` call emits `"callee_output_contract": null` in `main.ir.json`, even though the imported block returns `<current_branch>`.

Likely cause:

- `crates/glyph-core/src/emit_ir.rs` derives `callee_output_contract` by looking up a same-file `IrBlock`.
- Imported Tier-1 callees live only on `IrCall.callee_output_contract`, so the JSON path drops them.

Best fix:

- In `emit_ir.rs`, prefer the contract already attached to `IrCall`.
- Fall back to arena/block lookup only when the call-level contract is absent.
- Add an imported-callee IR JSON regression test.

This bug is also listed in the existing bug inventory, but the repro above verifies it in the current worktree.

## B14 - LSP and semantic-token coordinate conversion is likely wrong for UTF-16

Severity: medium. LSP uses UTF-16 character offsets by default, but Glyph spans and semantic tokens are byte-column based.

Evidence:

- `crates/glyph-lsp/src/convert.rs` maps Glyph byte columns directly to LSP `Position.character`.
- `crates/glyph-core/src/semantic_tokens.rs` notes that semantic token positions are byte columns and non-ASCII string literal contents may be off.
- `LineIndex::byte_offset` in `crates/glyph-core/src/span.rs` does not appear to clamp columns to line length despite its doc comment, so out-of-line cursor columns may map into following lines.

Best fix:

- Add UTF-16-aware position conversion against the source text for diagnostics, definitions, hover, and semantic tokens.
- Update `LineIndex::byte_offset` to clamp to the requested line's end.
- Add tests with non-ASCII characters before a referenced token and with cursor positions beyond line end.

## B15 - Condition string literals do not support escaped quotes

Severity: low-medium. Conditions containing escaped quotes tokenize incorrectly.

Evidence:

- `crates/glyph-core/src/condition.rs` has a TODO noting that escaped quotes are not supported.
- The condition tokenizer scans quoted strings until the next raw quote, so a literal such as `"a\"b"` truncates early.

Best fix:

- Teach `tokenize_condition` to honor backslash escapes inside quoted strings.
- Preserve spans for escaped string literals.
- Add tests for escaped quote, escaped backslash, and malformed trailing escape.
